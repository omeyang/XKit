# 技术文档书写标准（金标准）

**核心定位**：本文档是 `.specify` 目录下所有技术文档的**金标准（Golden Standard）**，定义文档质量的最高要求。

**适用范围**：`.specify` 目录下的所有技术文档

**强制执行范围**：
- `.specify/specs/*/analysis/` - 所有技术分析文档
- `.specify/specs/*/design/` - 所有设计文档（架构、方案、ADR等）
- `.specify/prior-art/` - 所有先验知识文档

**推荐遵守范围**：
- `.specify/specs/` - 规格文档（`spec.md`、`plan.md`、`tasks.md`）
- `.specify/memory/` - 项目记忆文档（`constitution.md` 等）
- `.specify/templates/` - 模板文档
- 各服务代码仓库中的技术文档（如 `docs/architecture/`）

## 1. 文档范围与目标

### 1.1 文档定位

本标准定义技术文档的书写规范，确保文档**职责唯一、简约准确、专业可读**，用于指导 go-idt 重构相关的分析文档和设计文档编写。

**适用文档类型**：
- 架构设计文档（Design Spec）
- 技术方案与算法设计
- 系统指标与性能分析
- 历史架构与技术债务分析

**不适用文档**（可参考但不强制）：
- 最终规格总览（spec.md）
- 实施计划（plan.md）
- 任务清单（tasks.md）
- 纯业务需求类文档

**文档边界与目的**：

本标准主要约束**技术分析与技术设计**类文档，不要求写出开发排期或详细测试用例，但要求：
- ✅ 技术方案必须基于现有代码和数据验证
- ✅ 设计决策必须考虑开发成本和可维护性
- ✅ 架构设计必须考虑性能、可测试性、可部署性
- ✅ 所有方案必须是可落地、可验证的

### 1.2 核心目标

| 目标 | 说明 | 检验标准 |
|------|------|----------|
| **统一编号** | 所有文档必须有唯一编号 | 文件名以编号开头（如 `01-xxx.md`）；拆分文档使用附录编号（如 `04-xxx.md` + `04A-xxx.md`） |
| **命名准确** | 文件名简短、准确、见名知义 | 文件名 ≤ 40 字符，使用 kebab-case |
| **职责唯一** | 每个文档只负责一个职责 | 文档标题能用一句话概括；无法用一句话概括则说明职责不清晰，需拆分 |
| **实质内容** | 文档内容必须有营养，干净利落 | 无冗余内容、无形式主义、无表演性质内容 |
| **时效准确** | 文档只记录最新状态 | 无历史演变描述；历史信息单独组织（如 ADR 目录） |
| **关联准确** | 所有引用和链接必须有效 | 文档路径、代码位置、数据来源均可验证 |
| **体系组织** | 文档按逻辑关系组织，符合阅读心智 | 有清晰的编号体系、目录结构、索引关系 |
| **代码溯源** | 剖析代码的文档必须标注代码基准 | 包含仓库路径、分支名称、提交 hash、分析日期 |
| **决策留痕** | 各阶段的决策必须记录，避免重复试错 | 有 decisions/ 或 adr/ 目录，记录已拒绝方案 |
| **简约精炼** | 避免冗余，突出核心 | 代码块 ≤ 5行，图表优先 |
| **控制篇幅** | 单个文档不超过 800 行 | 行数统计 ≤ 800，超过需拆分为"主文档 + 附录"（使用附录编号） |
| **专业表述** | 使用专业技术语言，杜绝口语化 | 无"可能"、"大概"、"用户说"等表达 |
| **真实可靠** | 基于实际代码和数据验证，不猜测臆想 | 所有结论有代码行号/数据库查询支撑 |
| **同架设计** | SaaS 和本地化使用同一代码架构，可配置切换 | 代码中体现配置开关，文档仅在差异处标注 |

### 原则 0：单一职责，职责清晰

**规则**：每个文档必须想清楚自己的职责，**只负责一个职责**。

**核心要求**：
1. **一句话测试**：文档标题能否用一句话准确概括？
2. **拆分判断**：如果文档包含多个独立主题，必须拆分
3. **内容一致性**：文档所有章节都围绕同一个职责展开

**检查清单**：
- [ ] 文档标题能否用一句话概括？
- [ ] 所有章节是否围绕同一职责展开？
- [ ] 是否包含了不属于本职责的内容？
- [ ] 是否存在"顺便讲了XX"的情况？

---

### 原则 1：图表优先，代码从简

**规则**：代码块不超过 **5 行**，超过则用 Mermaid 图表替代。

**核心要求**：
- 数据结构定义 → 使用 `classDiagram`
- 算法流程 → 使用 `flowchart`
- 系统架构 → 使用 `graph` 或 `C4Context`

**检查清单**：
- [ ] 代码块 ≤ 5 行（超过改用 Mermaid）
- [ ] 所有图表有明确标注
- [ ] 图表类型选择合理

---

### 原则 2：同架设计，差异标注

**规则**：代码架构必须统一，SaaS 和本地化通过配置切换，文档仅标注差异点。

**核心要求**：
1. **代码同架**：SaaS 和本地化使用同一套代码，避免分支维护
2. **配置切换**：通过配置文件或环境变量控制行为差异
3. **默认共同**：大多数功能是共同的，文档无需重复说明
4. **仅标注差异**：文档中只需明确标注 SaaS 和本地化的不同之处

**检查清单**：
- [ ] 代码架构统一（SaaS 和本地化使用同一代码）
- [ ] 标注了配置切换方式（配置文件/环境变量）
- [ ] 仅在差异处标注环境差异（非全量对比）

---

### 原则 3：专业术语，避免口语

**规则**：使用行业标准术语，删除口语化表达和非正式引用。

**核心要求**：
- 使用专业术语（QPS、P95、P99、CPU limit、Memory limit）
- 删除口语化（"可能"、"大概"、"应该"、"用户说"）
- 删除自问自答（FAQ、Q&A、"为什么..."、"如何..."）

**检查清单**：
- [ ] 删除了所有口语化表达（"可能"、"大概"、"用户说"）
- [ ] 删除了自问自答形式（包括FAQ、Q&A、"为什么..."、"如何..."等问答模式）
- [ ] 使用专业技术术语

---

### 原则 4：数据驱动，有据可查

**规则**：所有设计决策基于实际数据，标注数据来源。

**核心要求**：
- 标注数据来源（文档链接、MongoDB查询、Grafana监控）
- 记录采集时间和数据范围
- 提供查询方法或获取途径

**检查清单**：
- [ ] 所有数据标注了来源
- [ ] 标注了采集时间和数据范围
- [ ] 提供了查询方法或文档链接

---

### 原则 5：真实可靠，基于事实

**规则**：文档内容必须基于实际代码和数据，禁止猜测、臆想、假设。

**核心要求**：
1. **代码事实**：所有代码逻辑说明必须引用实际代码位置（文件:行号）
2. **数据事实**：所有数据结论必须提供数据来源（MongoDB查询/生产环境统计）
3. **明确边界**：知道的说知道，不知道的说不知道，不确定的明确标注
4. **禁用词汇**：删除所有不确定表达（"可能"、"大概"、"应该"、"估计"）

**检查清单**：
- [ ] 所有代码逻辑有代码位置引用（文件:行号）
- [ ] 所有数据有数据来源（MongoDB查询/生产环境统计）
- [ ] 不确定内容明确标注状态（✅基于代码/基于数据/基于分析 / ❓信息缺失）
- [ ] 无猜测、臆想、假设性表述
- [ ] 技术方案考虑了可落地性、可测试性、可维护性

---

### 原则 6：实质内容，拒绝形式主义

**规则**：文档必须干净利落、准确标准，每一行都有实质内容，不能为了写文档而写文档。

**核心要求**：
1. **有营养**：每个段落都传递有效信息，能解决实际问题
2. **无冗余**：删除重复内容、无意义的客套话、形式主义的章节
3. **无表演**：不写华而不实的"文档元数据"、"阅读指南"、"推荐路径"
4. **直击要害**：开门见山，直接讲技术方案，不绕弯子

**禁止的形式主义内容**：
- 联系方式、维护人、更新频率、问题反馈、文档版本、生成时间
- 总文档数、总阅读时间、推荐阅读路径、阅读建议
- 文档依赖关系图、文档成熟度矩阵
- 待办事项列表、实施计划、变更日志（这些应该在任务管理系统中）
- 场景示例、典型案例（除非是技术方案必需的说明）
- FAQ、Q&A、常见问题、问答模式

**检查清单**：
- [ ] 每个段落都有实质内容，不是为了写文档而写文档
- [ ] 删除了所有形式主义内容（联系方式、维护人、更新频率、文档版本等）
- [ ] 删除了所有表演性内容（阅读指南、推荐路径、FAQ、Q&A等）
- [ ] 删除了冗余的客套话和无意义的描述
- [ ] 每段内容通过"删除测试"（删除后技术方案仍完整则应删除）

---

### 原则 7：时效准确，只记录最新状态

**规则**：文档只关注当前最新的情况，不出现历史演变描述。

**核心要求**：
1. **只写现在**：文档描述的是"现在是什么样"，不写"曾经是什么样"
2. **历史单独管理**：有意义的历史信息单独组织在 ADR 目录或归档目录
3. **避免混淆**：防止读者看到过时信息，造成理解偏差

**历史信息的正确组织**：

| 信息类型 | 推荐位置 | 文件命名 |
|---------|---------|---------|
| 架构决策记录 | `.specify/specs/{feature}/design/adr/` | `001-decision-title.md` |
| 历史方案归档 | `.specify/specs/{feature}/design/archived/` | `old-plan-xxx.md` |
| 重构前的旧实现 | `.specify/prior-art/original-functionality/` | `old-xxx-implementation.md` |

**检查清单**：
- [ ] 文档只描述当前最新状态，不包含历史演变描述
- [ ] 历史信息单独组织在 ADR 目录或 archived 目录
- [ ] 不出现"以前是...，后来改成...，现在是..."的描述
- [ ] 如需说明历史原因，引用 ADR 文档而不是内联描述

---

### 原则 8：关联准确，引用必须有效

**规则**：文档中所有的引用、链接、路径必须是存在的、准确的、有效的。

**核心要求**：
1. **文档路径**：引用的文档必须真实存在
2. **代码位置**：引用的代码文件和行号必须准确
3. **数据来源**：引用的数据查询必须可复现
4. **及时更新**：当引用目标变化时，必须同步更新

**引用格式规范**：

| 引用类型 | 格式 | 示例 |
|---------|------|------|
| 文档引用 | `[描述](相对路径)` | `详见 [架构设计](./01-architecture.md)` |
| 代码位置 | `` `文件路径:行号` `` | `` `internal/belong/match.go:123-156` `` |
| 数据来源 | `[来源描述](附录位置)` | `MongoDB统计（见附录A1）` |
| 外部链接 | `[描述](完整URL)` | `[Grafana监控](http://grafana.example.com/dashboard/xxx)` |

**检查清单**：
- [ ] 所有文档引用 `[xxx](xxx.md)` 指向的文件都存在
- [ ] 所有代码位置引用（`file:line`）都准确有效
- [ ] 所有数据来源都有明确的查询方法或文档位置
- [ ] 没有模糊的引用（如"某个文档"、"之前说过"）
- [ ] 引用路径使用相对路径，格式规范

---

### 原则 9：体系组织，符合阅读心智

**规则**：文档必须被正确组织关联，有清晰的体系，符合人与AI的阅读心智历程，使用编号建立逻辑关系。

**核心要求**：
1. **逻辑顺序**：文档编号反映阅读顺序和逻辑依赖关系
2. **层级清晰**：主文档、附录、子主题有明确的编号规则
3. **索引完整**：目录下必须有 README.md 说明文档组织结构
4. **心智模型**：文档组织符合"总-分-总"或"问题-分析-方案"的认知规律

**编号体系规范**：

| 编号类型 | 格式 | 使用场景 | 示例 |
|---------|------|---------|------|
| 主文档 | `01-`, `02-`, ... | 独立主题，按逻辑顺序编号 | `01-overview.md`, `02-architecture.md` |
| 附录文档 | `01A-`, `01B-`, ... | 主文档的补充细节，使用字母后缀 | `04A-detailed-examples.md` |
| 子主题 | `01.1-`, `01.2-`, ... | 主题下的子主题（少用，优先拆分） | `03.1-cache-strategy.md` |
| ADR | `001-`, `002-`, ... | 架构决策记录，独立编号 | `adr/001-choose-mongodb.md` |

**特殊文档命名约定**（无需编号）：

| 文档类型 | 文件名 | 说明 |
|---------|--------|------|
| 目录索引 | `README.md` | 每个目录必须有 |
| 项目宪法 | `constitution.md` | 项目原则，固定名称 |
| 规格文档 | `spec.md` | Feature 规格，固定名称 |
| 实施计划 | `plan.md` | 实施计划，固定名称 |
| 任务清单 | `tasks.md` | 任务拆解，固定名称 |

**检查清单**：
- [ ] 所在目录有 `README.md` 索引文件
- [ ] 文档编号反映了逻辑阅读顺序
- [ ] 附录使用字母后缀（`04A-`, `04B-`）而不是新编号
- [ ] README.md 中列出了推荐阅读顺序
- [ ] 文档组织符合"总-分-总"或"问题-分析-方案"认知规律
- [ ] 历史文档单独组织在 `adr/` 或 `archived/` 目录

### 原则 10：代码溯源，明确基准

**规则**：所有剖析代码的文档必须明确标注代码仓库、分支、提交信息，防止多分支开发或 worktree 导致混乱。

**核心要求**：
1. **仓库路径**：明确代码仓库的绝对路径或相对项目根的路径
2. **分支名称**：明确分析的是哪个分支的代码
3. **提交信息**：记录提交 hash 和日期（推荐）
4. **分析日期**：记录文档分析代码的时间

**简化格式**（至少包含）：

```markdown
**代码基准**：`sangfor.com/xdr/go-idt` @ `feature-1030039-refactor` (`0d414bd`, 2025-01-18)
```

**代码位置引用格式**（包含分支信息）：

| 引用场景 | 格式 | 示例 |
|---------|------|------|
| 当前分支代码 | `` `文件:行号` `` | `` `internal/belong/match.go:123-156` `` |
| 其他分支代码 | `` `分支@文件:行号` `` | `` `master@internal/belong/match.go:123` `` |
| 历史版本代码 | `` `commit@文件:行号` `` | `` `0d414bd@internal/belong/match.go:123` `` |

**检查清单**：
- [ ] 文档开头或附录中标注了代码基准信息（仓库、分支、提交、日期）
- [ ] 包含仓库路径、分支名称、提交 hash
- [ ] 使用 worktree 时明确标注了工作树名称
- [ ] 代码位置引用时明确了是当前分支还是其他分支
- [ ] 提供了获取代码的 git 命令（可选）

### 原则 11：决策留痕，避免重复试错

**规则**：在开发的任何阶段（分析、规格、计划、设计、任务），一旦发现某个方向不可行，必须记录下来，避免 AI 或团队成员重复尝试错误路径。

**核心要求**：
1. **及时记录**：发现不可行的方向后，立即记录决策
2. **阶段适配**：不同阶段的决策记录在对应目录
3. **原因明确**：清楚说明为什么不可行
4. **通用命名**：使用 Decision Record (DR)，而不仅限于 Architecture Decision Record (ADR)

**决策记录类型**：

| 阶段 | 记录类型 | 目录位置 | 文件命名 | 说明 |
|------|---------|---------|---------|------|
| 分析阶段 | Analysis Decision Record | `.specify/specs/{feature}/01-analysis/decisions/` | `001-why-not-approach-a.md` | 分析时排除的方向 |
| 规格阶段 | Spec Decision Record | `.specify/specs/{feature}/02-spec/decisions/` | `001-rejected-requirement.md` | 需求阶段的取舍 |
| 设计阶段 | **Architecture Decision Record (ADR)** | `.specify/specs/{feature}/03-design/adr/` | `001-choose-cache-pattern.md` | **架构设计决策（保持传统命名）** |
| 计划阶段 | Planning Decision Record | `.specify/specs/{feature}/04-plan/decisions/` | `001-rejected-migration-path.md` | 计划时排除的实施路径 |

**ADR 全称说明**：

ADR = **Architecture Decision Record**（架构决策记录），是业界通用的架构设计决策标准。

在设计阶段使用 `adr/` 目录命名，保持与业界惯例一致。其他阶段使用更通用的 `decisions/` 命名。

**检查清单**：
- [ ] 每个阶段目录下都有 `decisions/` 或 `adr/` 目录（如有决策）
- [ ] 发现不可行的方向后立即记录决策
- [ ] 决策记录包含明确的拒绝理由和验证方法
- [ ] decisions 目录有 README.md 索引
- [ ] 决策记录标注了日期和决策者（人或 AI Session ID）
- [ ] 设计阶段使用 `adr/` 目录（保持 Architecture Decision Record 传统）
- [ ] 不在主文档中重复讨论已记录的拒绝方案（引用 DR 文档）

## 3. 文档检查流程

### 3.1 检查顺序

**编写完成后，按以下顺序检查**：

1. **原则 0-11 检查**：逐个原则对照各自的检查清单（第 2 节）
2. **书写标准检查**：参见 [附录C：书写标准详细说明](./01C-writing-standards-details.md)
3. **强制性级别检查**：参见 [附录D：强制性级别词汇约定](./01D-requirement-levels.md)
4. **中文写作规范检查**：参见 [附录E：中文技术写作规范](./01E-chinese-writing-style.md)

### 3.2 核心检查要点（快速自查）

**文档结构**：
- [ ] 文件名以编号开头，使用 kebab-case，≤ 40 字符
- [ ] 文档标题能用一句话概括（原则 0）
- [ ] 文档总行数 ≤ 800 行
- [ ] 章节层级不超过 4 级（H1-H4）

**内容质量**：
- [ ] 代码块 ≤ 5 行（原则 1）
- [ ] 删除口语化表达和自问自答（原则 3）
- [ ] 所有数据有来源、时间、查询方法（原则 4）
- [ ] 所有代码逻辑有位置引用（原则 5）
- [ ] 删除形式主义和表演性内容（原则 6）
- [ ] 只记录最新状态（原则 7）
- [ ] 所有引用和链接有效（原则 8）

**体系组织**：
- [ ] 所在目录有 README.md 索引（原则 9）
- [ ] 标注代码基准（仓库、分支、提交）（原则 10）
- [ ] 决策及时记录在 decisions/ 或 adr/ 目录（原则 11）

**排版规范**：
- [ ] 中英文、中文数字之间加空格
- [ ] 并列词语用顿号，时间范围用一字线，数值范围用浪纹线
- [ ] 强制性词汇使用**加粗**（"**必须**"、"**应该**"、"**可以**"）

# 附录D：强制性级别词汇约定

**上级文档**：[01-documentation-standards.md](./01-documentation-standards.md)

本附录定义文档中表达要求时的标准词汇，遵循 [RFC 2119](https://www.ietf.org/rfc/rfc2119.txt) 国际标准。

---

## 1. 为什么需要强制性级别约定？

**问题场景**：
- "建议使用缓存" vs "必须使用缓存" - 哪个是强制要求？
- "应该删除口语化" vs "可以删除口语化" - 执行力度如何？
- "推荐使用Mermaid" vs "优先使用Mermaid" - 是否可以不用？

**解决方案**：使用 RFC 2119 标准词汇，消除歧义。

---

## 2. 标准词汇定义

### 2.1 强制性要求

| 中文 | 英文 | 含义 | 示例 |
|------|------|------|------|
| **必须** | MUST / SHALL | 绝对要求，无例外 | 文档**必须**包含代码基准信息 |
| **禁止** | MUST NOT / SHALL NOT | 绝对禁止，无例外 | 文档**禁止**包含历史演变描述 |

**使用场景**：
- 安全要求（数据不能泄露）
- 架构约束（必须使用统一的错误处理）
- 规范遵守（代码块不超过5行）

**示例**：
```markdown
✅ 正确：所有技术文档**必须**标注代码基准信息（仓库、分支、提交hash）
✅ 正确：文档**禁止**使用口语化表达（"可能"、"大概"、"应该"）
```

### 2.2 强烈建议

| 中文 | 英文 | 含义 | 示例 |
|------|------|------|------|
| **应该** | SHOULD / RECOMMENDED | 强烈建议，除非有充分理由 | **应该**使用Mermaid图表替代长代码 |
| **不应该** | SHOULD NOT / NOT RECOMMENDED | 强烈不建议，除非有充分理由 | **不应该**在文档中包含待办事项 |

**使用场景**：
- 最佳实践（推荐但不强制）
- 代码风格（建议但可调整）
- 性能优化建议

**示例**：
```markdown
✅ 正确：表格列数**应该**≤5列（超过时考虑拆分）
✅ 正确：**不应该**使用H5、H6层级（需要时应拆分文档）
```

### 2.3 可选建议

| 中文 | 英文 | 含义 | 示例 |
|------|------|------|------|
| **可以** | MAY / OPTIONAL | 可选，实现者自由决定 | **可以**使用颜色高亮关键节点 |

**使用场景**：
- 锦上添花的功能
- 个性化选择
- 增强可读性的非必要措施

**示例**：
```markdown
✅ 正确：Mermaid图表**可以**使用颜色高亮关键节点
✅ 正确：代码位置引用**可以**提供git命令方便获取代码
```

## 3. 在技术文档中的应用

### 3.1 分级应用规则

**强制执行范围**（使用"必须"/"禁止"）：
- `.specify/specs/*/analysis/` - 技术分析文档
- `.specify/specs/*/design/` - 设计文档（架构、方案、ADR）
- `.specify/prior-art/` - 先验知识文档

**推荐遵守范围**（使用"应该"/"可以"）：
- `.specify/specs/` - 规格文档（spec.md、plan.md、tasks.md）
- `.specify/memory/` - 项目记忆文档
- `.specify/templates/` - 模板文档

### 3.2 示例对比

**❌ 模糊表述**：
```markdown
- 建议代码块不要太长
- 最好使用Mermaid图表
- 推荐标注代码位置
```

**✅ 明确表述**：
```markdown
- 代码块**必须**≤5行（强制要求）
- **应该**使用Mermaid图表替代超过5行的代码（强烈建议）
- **可以**标注git命令方便代码获取（可选增强）
```

## 4. 现有原则的强制性级别标注

基于现有12个核心原则，标注强制性级别：

| 原则 | 强制性级别 | 关键要求 |
|------|-----------|---------|
| **原则0：单一职责** | MUST | 文档**必须**职责唯一，标题**必须**能用一句话概括 |
| **原则1：图表优先** | MUST | 代码块**必须**≤5行，超过**必须**改用Mermaid |
| **原则2：同架设计** | MUST | **必须**使用同一代码架构，**禁止**分支维护 |
| **原则3：专业术语** | MUST | **必须**删除口语化，**禁止**自问自答 |
| **原则4：数据驱动** | MUST | 所有数据**必须**标注来源 |
| **原则5：真实可靠** | MUST | **必须**基于实际代码，**禁止**猜测臆想 |
| **原则6：实质内容** | MUST | **必须**删除形式主义内容 |
| **原则7：时效准确** | MUST | **必须**只记录最新状态，**禁止**历史演变描述 |
| **原则8：关联准确** | MUST | 所有引用**必须**有效可验证 |
| **原则9：体系组织** | MUST | 所在目录**必须**有README.md索引 |
| **原则10：代码溯源** | MUST | **必须**标注代码基准（仓库、分支、提交） |
| **原则11：决策留痕** | MUST | 发现不可行方向**必须**立即记录决策 |

**书写标准的强制性级别**：

| 标准 | 强制性级别 | 要求 |
|------|-----------|------|
| **表格列数** | SHOULD | **应该**≤5列（超过考虑拆分） |
| **章节层级** | SHOULD NOT | **不应该**使用H5、H6（需要时应拆分文档） |
| **文档长度** | MUST | **必须**≤800行（超过拆分为主文档+附录） |
| **图表标注** | SHOULD | **应该**使用颜色高亮关键节点 |
| **代码高亮** | MUST | **必须**标注语言类型 |

## 5. 检查清单

### 强制性表述检查

- [ ] 绝对要求使用"**必须**"（MUST）
- [ ] 绝对禁止使用"**禁止**"（MUST NOT）
- [ ] 强烈建议使用"**应该**"（SHOULD）
- [ ] 可选建议使用"**可以**"（MAY）
- [ ] 避免模糊表述（"建议"、"推荐"、"最好"、"尽量"）

### 例外情况说明

如果需要偏离"应该"级别的要求，**必须**说明理由：

```markdown
**偏离说明**：虽然表格列数**应该**≤5列，但本表格包含7列是因为：
- 这是性能基线数据的完整记录
- 拆分表格会破坏数据的对比性
- 已经在README中提供了简化版视图
```

## 6. RFC 2119 使用指南（重要）

**完整 RFC 2119 文档**：`/root/code/ai/tools/2119/rfc2119.txt`（本地资源，按需查阅）

### 6.1 谨慎和节制原则

**RFC 2119 原文强调**：
> "Imperatives of the type defined in this memo **must be used with care and sparingly**."
>
> （本备忘录中定义的这类命令词**必须谨慎和节制地使用**。）

**核心要求**：
- ✅ **仅在必要时使用**：互操作性要求、防止安全漏洞/数据损坏
- ❌ **避免过度使用**：不影响互操作性的实现细节不应强制

**示例**（合理分级 vs 过度使用）：
```markdown
✅ 函数名**应该**使用驼峰命名（代码风格，可调整）
✅ 代码块**必须**有空行分隔（Markdown 渲染要求，影响可读性）

❌ 函数名**必须**使用驼峰命名（过度强制实现细节）
```

### 6.2 安全考虑（关键）

**RFC 2119 安全警告**：
> "The effects on security of not implementing a MUST or SHOULD, or doing something the specification says MUST NOT or SHOULD NOT be done **may be very subtle**."
>
> （不实施 MUST 或 SHOULD，或做规范说 MUST NOT 或 SHOULD NOT 做的事情，对安全性的影响**可能非常微妙**。）

**文档作者的责任**：涉及安全性时，**必须**说明违反后果、原因和检查方法。

**示例**（安全要求的完整说明）：
```markdown
✅ 正确示例：
API 响应**禁止**包含敏感信息（密码、Token）。

**违反后果**：敏感信息泄露导致攻击者可直接登录系统。
**检查方法**：
- 代码审查：搜索 API 返回结构中的敏感字段
- 自动化测试：Mock API 响应，断言不包含敏感字段
- 渗透测试：使用 Burp Suite 拦截响应检查泄露
```

### 6.3 标准引用声明

**在文档开头包含以下声明**（可选但推荐）：

```markdown
## 关键词约定

本文档中的关键词"**必须**"、"**禁止**"、"**应该**"、"**不应该**"和"**可以**"
按照 [RFC 2119](https://www.ietf.org/rfc/rfc2119.txt) 中的描述进行解释。
```

**说明**：英文文档的标准声明格式见 `/root/code/ai/tools/2119/rfc2119.txt` 第 10-12 行。

### 6.4 大小写约定

**英文文档**：关键词**必须大写**，以区分规范性关键词和普通词汇

- ✅ `The implementation MUST validate input.`（规范性要求）
- ❌ `The implementation must validate input.`（普通英语，非规范性）

**中文文档**：使用**加粗**强调关键词

- ✅ `实现**必须**验证输入。`
- ❌ `实现必须验证输入。`（不够醒目）

### 6.5 语境修饰

**注意**：这些关键词的效力受文档类型的影响

| 文档类型 | 强制性 | 说明 |
|---------|-------|------|
| 国际标准 (RFC, ISO) | 最高 | 必须严格遵守 |
| 行业规范 (公司标准) | 高 | 强烈建议遵守 |
| 最佳实践指南 | 中 | 参考建议 |
| 草案/讨论稿 | 低 | 仅供参考 |

**本文档定位**：公司内部技术文档规范（高强制性）

## 7. 与业界标准的对应

| RFC 2119 | 本文档 | 强制性 | 说明 |
|----------|--------|--------|------|
| MUST / REQUIRED / SHALL | 必须 | 100% | 绝对要求 |
| MUST NOT / SHALL NOT | 禁止 | 100% | 绝对禁止 |
| SHOULD / RECOMMENDED | 应该 | ~80% | 强烈建议，有充分理由可偏离 |
| SHOULD NOT / NOT RECOMMENDED | 不应该 | ~80% | 强烈不建议 |
| MAY / OPTIONAL | 可以 | 0% | 完全可选 |

## 设计原则

两个消息队列包遵循统一设计模式：

1. **透明封装**：通过 `Producer()` / `Consumer()` / `Client()` 暴露底层 API
2. **可选增强**：基础版 + TracingProducer/TracingConsumer（装饰器模式）
3. **DLQ 支持**：死信队列与重试策略（集成 xretry）
4. **链路追踪**：消息级别的 trace context 注入/提取

审查时需验证两个包是否一致遵循上述模式，以及差异是否有充分理由。

## 模块职责与审查要点

### xkafka — Kafka 客户端封装

**职责**：Producer/Consumer 基础封装、TracingProducer/TracingConsumer、ConsumerWithDLQ。

**重点审查**：

#### 消息投递保证
- [ ] **At-least-once 语义**：`enable.auto.offset.store=false` 是否正确配置？手动 commit 时机？
- [ ] **Offset 管理**：消息处理成功后才 commit，还是消费到就 commit？
- [ ] **Consumer 退出**：Close 时是否等待当前消息处理完成？uncommitted offset 是否会导致重复消费？
- [ ] **Producer 可靠性**：`acks` 配置是否合适（acks=all vs acks=1）？默认值是否安全？

#### 分区与并发
- [ ] **分区分配**：Consumer 的分区分配策略（Range/RoundRobin/Sticky）是否可配置？
- [ ] **Rebalance 处理**：Rebalance 期间消息处理是否正确暂停和恢复？
- [ ] **并发消费**：是否支持多 goroutine 消费同一 consumer group？线程安全性？

#### DLQ（死信队列）
- [ ] **重试次数**：消息最大重试次数是否可配置？超过重试次数后是否发送到 DLQ topic？
- [ ] **重试退避**：退避策略是否集成 xretry.BackoffPolicy？
- [ ] **DLQ 消息格式**：原始消息 + 错误信息 + 重试元数据是否完整保留？
- [ ] **DLQ topic 命名**：是否有命名规范（如 `{original-topic}-dlq`）？

#### 追踪
- [ ] **TracingProducer**：是否在消息 header 中注入 traceparent/tracestate？
- [ ] **TracingConsumer**：是否从消息 header 提取 trace context 并创建子 span？
- [ ] **span 属性**：topic、partition、offset、consumer group 是否作为 span 属性？

#### 错误处理
- [ ] **Kafka 错误分类**：哪些 Kafka 错误可重试（网络超时）、哪些不可重试（认证失败）？
- [ ] **连接断开恢复**：Broker 断开后是否自动重连？重连期间消息是否丢失？
- [ ] **Producer 发送失败**：异步发送失败的回调是否可靠？是否有消息丢失风险？

### xpulsar — Pulsar 客户端封装

**职责**：Client 包装器、TracingProducer/TracingConsumer、DLQBuilder、NackBackoff。

**重点审查**：

#### 订阅模式
- [ ] **模式覆盖**：是否支持 Exclusive/Shared/Failover/KeyShared 全部四种？
- [ ] **KeyShared 正确性**：相同 key 的消息是否保证路由到同一 consumer？hash 算法是否稳定？
- [ ] **模式切换**：已有订阅的模式切换是否有保护（Pulsar 不允许变更已有订阅的类型）？

#### DLQ 与重试
- [ ] **DLQBuilder**：DLQ policy 配置是否完整（maxRedeliverCount, deadLetterTopic）？
- [ ] **NackBackoff**：是否正确适配 xretry.BackoffPolicy？退避时间计算是否准确？
- [ ] **消息确认**：Ack 和 Nack 的语义是否正确？Nack 后是否立即重投递还是按退避等待？

#### Schema 管理
- [ ] **Schema 兼容性**：是否支持 Avro/JSON/Protobuf Schema？Schema 演进（evolution）策略？
- [ ] **Schema 验证**：消息是否在发送时进行 Schema 验证？无效消息是否拒绝？

#### 多租户
- [ ] **Pulsar 原生多租户**：是否利用 Pulsar 的 tenant/namespace 层级？与 xtenant 的集成？
- [ ] **topic 命名**：persistent://tenant/namespace/topic 的命名是否规范？

#### 追踪
- [ ] **TracingProducer/Consumer**：与 xkafka 相同的追踪要求
- [ ] **span 属性**：topic、subscription name、message ID 是否作为 span 属性？

## 跨包一致性检查

- [ ] xkafka 和 xpulsar 的 DLQ 重试策略是否都使用 xretry？接口是否一致？
- [ ] Tracing 是否都通过 xmetrics/xtrace 接口？是否直接依赖 OTel SDK？
- [ ] 消息头中的 trace propagation 格式是否与 xtrace 的 W3C 格式一致？
- [ ] 两个包的 Options 模式命名是否统一？
- [ ] 错误类型是否统一（如 `ErrConsumerClosed`, `ErrProducerClosed`）？
- [ ] metrics 指标命名是否遵循 `xkit.mq.*` 前缀？counter/histogram 选择是否一致？
